------ ABOUT FOLDER STRUCTURE ------
In "modules" folder we have folder for each feature and in each we have controller, service, respository(model) etc...

The Architecture: This project uses a Domain-Driven (Modular) Architecture, grouping files by business feature (e.g., Auth, Task, Sprint) rather than by technical role, keeping everything related to one domain in a single, isolated folder. 
Why we avoid traditional MVC: Standard MVC groups files by technical role (putting every controller for the entire app into one massive folder), which creates Git merge conflicts, makes code navigation a nightmare, and tightly couples the system.
Why this is better: It is built for enterprise scalability; feature isolation makes it incredibly easy to maintain, add new developers without stepping on toes, and eventually break specific modules (like Notifications) out into their own independent microservices.
The Disadvantages: It has a steeper learning curve, it is absolute overkill for small or simple applications, and it requires strict discipline to ensure modules don't accidentally become deeply entangled with one another.
